Enterprise Alert Platform

统一告警，智能运维

SentinelX 是一款企业级、多租户、高可用的综合告警平台。 将 Prometheus、云监控、Zabbix 等异构告警源统一接入， 通过规则引擎实现智能路由、去重聚合与多渠道通知，让运维团队从告警风暴中解脱。

AI Powered

AI 智能增强

在告警详情页一键调用 LLM，辅助值班人员快速理解告警、生成通知文案与处置建议。 按租户独立配置模型与 API Key（AES 加密存储），任务通过 PGMQ alerts_ai 队列异步执行，不阻塞主流程。

🔬

根因分析

结合告警标题、标签、指标值与触发上下文，分析可能原因并给出排查方向，输出结构化 Markdown 报告。

POST /api/v1/alerts/{id}/analyze
✨

内容润色

将原始告警润色为适合通知渠道发送的文案，支持 formal / simple / friendly 三种风格，可配合通知模板使用。

POST /api/v1/alerts/{id}/polish?style=formal
🛠️

建议操作

参考同 alert_key 的历史告警记录，推荐可执行的排查与处置步骤，帮助新手值班人员快速上手。

POST /api/v1/alerts/{id}/suggest-actions
📈

影响预测

评估告警若未及时处理的潜在业务与服务影响，辅助判断优先级与是否需要立即升级。

POST /api/v1/alerts/{id}/predict-impact

支持的 LLM 提供商（可在「系统设置 → AI 设置」中切换）：

OpenAI DeepSeek Anthropic Claude 通义千问 Qwen Custom（OpenAI 兼容）
告警详情页发起 → API 返回 202 + task_id → PGMQ alerts_ai → AIWorker 调用 LLM → 前端轮询展示结果

🔐 租户级配置

每个租户独立配置 provider、model 与 API Key，密钥 AES 加密存于 tenant.config.ai，互不影响。

📝 自定义 Prompt

四大模块（analyze / polish / suggest_actions / predict_impact）均支持自定义 System Prompt，覆盖内置默认即可生效。

⚡ 异步非阻塞

AI 任务与告警主链路解耦，LLM 响应较慢时不影响告警接收、规则匹配与通知发送。

🎛️ 按需启用

租户超管在 AI 设置中开启后，告警详情页才显示 AI 操作入口；未配置则功能完全隐藏。

Alert Policies

去重 · 抑制 · 聚合

三大告警治理策略，按需配置、未配则跳过。每条告警按「指纹生成 → 去重 → 抑制 → 聚合 → 规则路由 → 通知」顺序流转， 各阶段结果可在 Trace 诊断 中逐步回溯。

🔄

去重（Dedup）

在指定时间窗口内，对满足条件的重复告警进行丢弃，避免同一故障引发告警风暴。命中后状态标记为「已去重」。

  • 指纹模式：按 alert_key 等字段生成指纹去重
  • 条件模式：AND/OR 标签条件 + 时间窗口
  • 规则预览 POST /rules/preview-dedup
  • 未配置去重规则时，每条消息正常流转
🚫

抑制（Suppress）

在维护窗口或规则命中时阻止告警继续通知，适用于发布变更、计划内维护等场景，避免无效打扰。

  • 维护窗口：按时间段自动抑制，同步 Redis 快速判断
  • 抑制规则：标签条件匹配 + 按时解除
  • 规则禁用时自动跳过该阶段
  • Trace 中区分「未触发 / 已抑制 / 规则关闭」
📦

聚合（Aggregate）

将多条同类告警合并为策略聚合组，列表以虚拟指纹行展示，子告警标记 aggregated，值班界面更清爽。

  • 条件聚合：按 namespace、source 等字段分组
  • 指纹视图 + aggregated-members 查成员
  • 规则预览 POST /rules/preview-aggregate
  • 默认隐藏 aggregated 子告警，可参数展开

另有告警升级策略：未确认 firing 告警按 5 → 10 → 30 → 60 分钟逐级升级，由 EscalationWorker 异步处理。

Observability

Trace 全链路诊断

每条告警分配唯一 12 位 Trace ID，贯穿接入、去重、抑制、聚合、规则匹配到通知队列的全流程。 值班同学可在诊断页输入 Trace ID，或从告警详情一键跳转，快速回答「这条告警为什么没通知？」。

12 位

唯一 Trace ID

告警从接入到处理结束全程携带同一 ID，支持跨日志、跨模块关联排查。

GET /api/v1/alerts/diagnose/{trace_id}
逐步

可视化诊断页

前端诊断页展示 flow_steps 时间线：每步标题、状态、原因与详情，无需翻后端日志。

/diagnose?trace=xxx
1

去重检查 passed

未命中 / 未配置规则 → 跳过；命中 → blocked，告警丢弃并标记「已去重」。

2

抑制检查 skipped

维护窗口或抑制规则命中 → blocked；规则已禁用 → 跳过；未命中 → 继续流转。

3

聚合检查 aggregated

加入现有组 → aggregated；创建新组 → new_group；无规则 / 不匹配 → 跳过。

4

规则匹配 success

命中 N 条路由规则并绑定通知渠道；无命中 → 使用默认渠道 fallback。

5

通知队列 success

写入 PGMQ alerts_notify；60 秒内同指纹已通知则 dedup_skipped 跳过重复发送。

诊断步骤 常见状态 含义
去重检查passed / 跳过未命中去重，或无规则配置
去重检查blocked触发去重，告警被丢弃
抑制检查blocked维护窗口或抑制规则命中
聚合检查aggregated加入已有聚合组
通知队列dedup_skipped60 秒内同指纹已通知，跳过发送
Multi-Tenant

多租户切换 · 注册审批

支持集团多团队共用同一平台：用户与租户 N:M 关联，前端一键切换上下文；新用户注册需管理员审批后方可登录，兼顾灵活性与安全。

👥

多租户快速切换

一个用户可同时属于多个租户（如集团下各事业部），登录后在顶栏选择当前租户，所有告警、规则、渠道数据按租户隔离。

  • is_current 标记当前活跃租户
  • is_primary 标记用户主租户
  • POST /api/v1/auth/switch-tenant
  • GET /api/v1/auth/tenants 列出我的租户
  • 系统管理员（is_system）拥有全局权限
✅

用户注册审批

开放注册模式下，新用户提交注册后处于待审批状态，无法登录。系统管理员审核通过并分配角色/租户后，账号方可正常使用。

  • 注册后 is_approved=false，拒绝登录
  • 管理员 POST /admin/users/{id}/approve
  • 可分配系统级或租户级角色
  • 租户管理员管理本租户内用户
  • 拒绝审批则账号保持无效

注册与审批流程：

用户注册 → 待审批 is_approved=false → 管理员审批 → 分配角色 / 租户 → 正常登录

审批拒绝 → 账号无效，流程终止。

Performance

毫秒级数据查询

告警标签结构动态多变，SentinelX 在 PostgreSQL 层充分利用 JSONB、GIN 索引与窗口函数， 避免应用层全表扫描，支撑大规模告警场景下的实时检索与聚合。

ms 级

JSONB + GIN 索引

告警 labels 以 JSONB 存储，GIN 索引支持任意标签键值的等值与包含查询，无需预定义 Schema 或额外表结构。

动态标签 规则条件匹配 字段值下拉
SQL 层

CTE + 窗口函数

指纹视图、策略聚合、去重预览均在数据库层通过 CTE 与窗口函数完成，减少 Python 侧内存聚合与多次往返。

指纹视图 聚合预览 去重预览
零中间件

PGMQ 原生队列

告警接收与通知发送均通过 PostgreSQL 原生消息队列异步解耦，无需部署 Kafka / RabbitMQ 等额外组件。

alerts_raw alerts_notify alerts_escalation

Deployment

部署方式

从本地开发到生产集群，提供多种部署选项。

复杂度 · 低

Docker Compose

一键启动完整服务栈（后端、前端、PostgreSQL、Redis），适合开发测试与小规模生产。支持仅启动基础设施供本地开发。

提供完整 Helm Chart，支持 HPA 自动扩缩、Ingress、外置数据库等生产级配置，适合大规模部署。

本地开发
